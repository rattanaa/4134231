# SRS: ระบบตรวจเช็คค่าน้ำประปาหมู่บ้าน

## 1. บทนำ

### 1.1 วัตถุประสงค์
เอกสารนี้เป็นข้อกำหนดซอฟต์แวร์ (Software Requirements Specification) สำหรับระบบเว็บแอปพลิเคชันตรวจเช็คค่าน้ำประปาหมู่บ้าน เพื่อเป็นแนวทางการพัฒนา ทดสอบ และส่งมอบระบบ

### 1.2 ขอบเขตของระบบ
ระบบเป็นเว็บแอปที่ให้ผู้ใช้งานลงชื่อเข้าใช้ ดูข้อมูลที่อยู่ บันทึก/ตรวจสอบค่าน้ำและยอดเรียกเก็บสำหรับผู้ใช้น้ำในหมู่บ้าน และมีหน้าจอสำหรับผู้ดูแลในการจัดการข้อมูลผู้ใช้งานและรายงาน

## 2. คำจำกัดความ
- ผู้ใช้น้ำ: ผู้อยู่อาศัยที่มีสิทธิ์ดูค่าน้ำของตน
- ผู้ดูแล (Admin): บุคคลที่จัดการผู้ใช้งาน ข้อมูลมิเตอร์ และรายงาน

## 3. คำอธิบายภาพรวม

### 3.1 มุมมองของผลิตภัณฑ์
ระบบเป็นเว็บแอปแบบ Responsive ใช้งานได้ทั้งบนมือถือและเครื่องพีซี ผ่านเบราว์เซอร์สมัยใหม่

### 3.2 ประเภทผู้ใช้งาน
- Resident (ผู้ใช้น้ำ)
- Admin (ผู้ดูแลระบบ)

### 3.3 ข้อจำกัดแวดล้อมการทำงาน
- ต้องรองรับการใช้งานบนเบราว์เซอร์หลัก (Chrome, Edge, Safari/มือถือ)
- ใช้งานได้ทั้งมือถือและพีซี

## 4. ข้อกำหนดเชิงฟังก์ชัน (Functional Requirements)

FR-1: การยืนยันตัวตน
- ระบบต้องให้ผู้ใช้งานลงชื่อเข้าใช้ด้วยบัญชีและรหัสผ่าน

FR-2: ข้อมูลโปรไฟล์และที่อยู่
- ผู้ใช้งานสามารถดูและแก้ไขข้อมูลที่อยู่ และข้อมูลติดต่อของตน

FR-3: บันทึก/ตรวจเช็คค่าน้ำ
- ผู้ดูแล/ผู้ปฏิบัติงานสามารถบันทึกค่ามิเตอร์รายเดือน
- ผู้ใช้น้ำสามารถดูค่าน้ำที่บันทึกและยอดเรียกเก็บ

FR-4: ประวัติและใบแจ้งหนี้
- ผู้ใช้น้ำสามารถดูประวัติการอ่านมิเตอร์และใบแจ้งค่าน้ำย้อนหลัง

FR-5: การจัดการผู้ใช้งาน (Admin)
- ผู้ดูแลเพิ่ม แก้ไข หยุดการใช้งานบัญชีผู้ใช้น้ำ

FR-6: รายงานและการส่งออก
- ระบบให้รายงานยอดการใช้น้ำตามช่วงเวลา และสามารถส่งออก CSV/PDF สำหรับผู้ดูแล

FR-7: การแจ้งเตือน (โต้แย้งเป็นทางเลือก)
- ระบบสามารถแจ้งเตือนผู้ใช้งานเมื่อมีใบแจ้งค่าน้ำใหม่ (ผ่าน UI หรืออีเมล หากมี)

## 5. ข้อกำหนดเชิงอำนวยการ (Non-Functional Requirements)

NFR-1: การตอบสนองและประสิทธิภาพ
- หน้าเว็บหลักต้องโหลดภายใน 3 วินาทีบนเครือข่ายมือถือทั่วไป

NFR-2: ความสามารถใช้งาน (Usability)
- UI ต้องเป็น Responsive และใช้งานง่ายบนหน้าจอมือถือ

NFR-3: ความปลอดภัย
- ข้อมูลการเข้าสู่ระบบต้องเข้ารหัส (HTTPS)
- รหัสผ่านต้องเก็บอย่างปลอดภัย (hash + salt)

NFR-4: ความพร้อมใช้งานและสำรองข้อมูล
- ระบบควรมีนโยบายสำรองข้อมูลรายวันและสามารถกู้คืนข้อมูลได้

NFR-5: การรองรับภาษา
- อินเทอร์เฟซหลักต้องเป็นภาษาไทย (รองรับภาษาอื่นเป็นทางเลือก)

## 6. แบบจำลองระบบ (Use Cases / Screens)
- UC-1 ลงชื่อเข้าใช้ / ออกจากระบบ
- UC-2 ดู/แก้ไขโปรไฟล์และที่อยู่
- UC-3 ดูยอดค่าน้ำปัจจุบันและประวัติ
- UC-4 ผู้ดูแลบันทึกค่ามิเตอร์
- UC-5 ผู้ดูแลสร้างรายงาน

หน้าจอหลักที่ต้องมี
- หน้า Login
- Dashboard (ผู้ใช้น้ำ: ยอดปัจจุบัน/ประวัติ)
- หน้าโปรไฟล์/ที่อยู่
- หน้า Manager (Admin): ผู้ใช้, มิเตอร์, รายงาน

## 7. ข้อจำกัด (Constraints)
- ระบบต้องทำงานผ่านเว็บ ไม่ต้องติดตั้งแอปเนทีฟ
- ต้องรองรับการเข้าถึงผ่านมือถือและพีซี

## 8. สมมติฐาน
- ผู้ใช้งานมีสมาร์ทโฟนหรือพีซีและใช้อินเทอร์เน็ต
- ระบบชำระเงินอาจเป็นภายนอก (ไม่รวมเป็นภายในขอบเขตเริ่มต้น)

## 9. เกณฑ์การรับมอบ (Acceptance Criteria)
- ผู้ใช้น้ำสามารถลงชื่อเข้าใช้และดูยอดค่าน้ำปัจจุบัน
- ผู้ดูแลสามารถบันทึกการอ่านมิเตอร์และข้อมูลนั้นแสดงให้ผู้ใช้น้ำเห็น
- UI แสดงผลได้ถูกต้องบนมือถือและพีซี

## 10. ภาคผนวก
- ข้อมูลที่จำเป็นเพิ่มเติม เช่น รูปแบบฐานข้อมูล ฟิลด์ผู้ใช้งาน จะกำหนดในขั้นออกแบบรายละเอียดถัดไป

# Software Requirements Specification (SRS)

## โครงการ
สร้างเว็บแอปสำหรับการจัดการและตรวจเช็คค่าน้ำประปาของหมู่บ้าน

## ที่มาของข้อมูล
ข้อกำหนดต้นทาง: [readme.md](readme.md#L1-L4)

## วัตถุประสงค์
- ให้ผู้ใช้น้ำ (ผู้อยู่อาศัย) ลงชื่อเข้าใช้ ดูที่อยู่และข้อมูลการใช้น้ำ
- ให้เจ้าหน้าที่/ผู้ดูแลระบบบันทึกและตรวจสอบค่าน้ำ รายงาน และแจ้งเตือน
- ระบบต้องรองรับทั้งการใช้งานบนมือถือและ PC (responsive)

## ขอบเขตของระบบ
- จัดการบัญชีผู้ใช้ (ลงทะเบียน, เข้าสู่ระบบ, ลืมรหัส)
- เก็บข้อมูลที่อยู่ของผู้ใช้น้ำ
- บันทึก/อัปเดตการอ่านมาตรค่าน้ำ (meter readings)
- แสดงประวัติการใช้น้ำและค่าน้ำที่ต้องชำระ
- ระบบรายงานสำหรับเจ้าหน้าที่ (เช่น ยอดรวม, ผู้ค้างชำระ)
- การแจ้งเตือน (e.g., แจ้งเตือนค่าน้ำครบกำหนด)

## ผู้ใช้งาน (User Classes)
- ผู้ใช้น้ำ: ลงชื่อเข้าใช้ ดูข้อมูลการใช้น้ำและที่อยู่
- เจ้าหน้าที่/ผู้ดูแลระบบ: จัดการผู้ใช้ บันทึกการอ่าน และสร้างรายงาน
- ผู้ดูแลระบบระบบ: จัดการสิทธิ์และตั้งค่าระบบ

## ข้อกำหนดเชิงฟังก์ชัน (Functional Requirements)
- FR1: ผู้ใช้สามารถลงทะเบียนด้วยชื่อ ที่อยู่ และข้อมูลติดต่อ
- FR2: ผู้ใช้สามารถเข้าสู่ระบบและออกจากระบบได้
- FR3: ผู้ใช้สามารถดูที่อยู่และประวัติการใช้น้ำของตน
- FR4: เจ้าหน้าที่สามารถบันทึก/แก้ไขการอ่านมาตรวัดค่าน้ำสำหรับผู้ใช้แต่ละราย
- FR5: ระบบต้องคำนวณยอดค่าน้ำตามอัตราที่กำหนดและแสดงรายการที่ต้องชำระ
- FR6: เจ้าหน้าที่สามารถสร้างรายงานสรุปการใช้น้ำและผู้ค้างชำระ
- FR7: ระบบส่งแจ้งเตือนเมื่อค่าน้ำถึงกำหนดหรือค้างชำระ

## ข้อกำหนดเชิงไม่ทำงาน (Non-Functional Requirements)
- NFR1: รองรับการใช้งานบนมือถือและ PC (responsive UI)
- NFR2: ความปลอดภัย: รหัสผ่านต้องถูกเก็บแบบ hash (เช่น bcrypt)
- NFR3: ประสิทธิภาพ: หน้าแดชบอร์ดโหลดภายใน 3 วินาทีสำหรับข้อมูลปกติ
- NFR4: ความพร้อมใช้งาน: ระบบต้องพร้อมใช้งาน 99% ต่อเดือน
- NFR5: สำรองข้อมูล: มีนโยบายสำรองข้อมูลประจำวัน

## ข้อมูลและโมเดลข้อมูล (Data Requirements)
- ข้อมูลผู้ใช้: user_id, ชื่อ, เบอร์โทร, อีเมล, รหัสผ่าน(เข้ารหัส), ที่อยู่
- ข้อมูลการอ่านมาตรวัด: reading_id, user_id, วันที่อ่าน, ค่าน้ำหน่วย, ผู้บันทึก
- ข้อมูลการชำระเงิน: payment_id, user_id, วันที่ชำระ, จำนวนเงิน, วิธีการ

## อินเทอร์เฟซภายนอก
- UI: เว็บ responsive ใช้งานผ่านเบราว์เซอร์
- อีเมล: ระบบส่งอีเมลแจ้งเตือน (SMTP)

## กรณีใช้งานหลัก (Key Use Cases)
- UC1: ลงทะเบียนผู้ใช้น้ำ
- UC2: ผู้ใช้น้ำเข้าสู่ระบบและดูประวัติ
- UC3: เจ้าหน้าที่บันทึกการอ่านมาตรวัด
- UC4: ระบบคำนวณและแสดงยอดค่าน้ำ
- UC5: เจ้าหน้าที่สร้างรายงานเดือน/ปี

## ข้อจำกัด (Constraints)
- ต้องทำงานผ่านเว็บเบราว์เซอร์สมัยใหม่
- ต้องรองรับการแสดงผลบนหน้าจอมือถือและเดสก์ท็อป

## สมมติฐาน (Assumptions)
- ผู้ใช้งานมีการเชื่อมต่ออินเทอร์เน็ต
- เจ้าหน้าที่มีสิทธิ์เข้าถึงข้อมูลที่เกี่ยวข้อง

## เกณฑ์ยอมรับ (Acceptance Criteria)
- ระบบต้องให้ผู้ใช้ลงทะเบียนและล็อกอินได้สำเร็จ
- เจ้าหน้าที่สามารถบันทึกการอ่านและข้อมูลปรากฏในประวัติ
- หน้าต่างๆ แสดงผลอย่างถูกต้องบนมือถือและ PC
